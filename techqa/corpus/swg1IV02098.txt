   command then restart qmgrs.
   
   
    
   
   

PROBLEM SUMMARY
 *  ****************************************************************
   USERS AFFECTED:
   A user may be affected by this problem if a great many channel
   start requests fail, and the fix for APAR IZ75682 [http://www-01.ibm.com/support/docview.wss?uid=swg1IZ75682] (a shared
   memory leak) is not in place. The problem could also be
   encountered with "mqm" applications failing to connect (instead
   of channel start failures).
   
   Platforms affected:
   All Unix
   
   ****************************************************************
   PROBLEM SUMMARY:
   There is a theoretical, and rare, circumstance in the
   management of queue manager shared memory where a lock may be
   held during shared memory extension, then may be required again
   for extension of a further aspect. That causes a deadlock on the
   thread taking out the lock.
   
   
    
   
   

PROBLEM CONCLUSION
 *  The lock handling of queue manager shared memory extension has
   been corrected to avoid the possibility of attempting to take
   out a lock that is already held.
   
   It is unlikely in practice that this problem could be
   encountered subsequent to the fix of APAR IZ75682 [http://www-01.ibm.com/support/docview.wss?uid=swg1IZ75682]. That APAR
   fixes a small shared memory leak that triggered the need to
   extend queue manager shared memory.
