APAR STATUS
 * CLOSED AS PROGRAM ERROR.
    
   
   

ERROR DESCRIPTION
 *  In certain cases, if you export a release of a base
   portal (with XMLAccess/ExportRelease.xml), then export a
   release of one virtual portal, then a subsequent export
   of a release from another virtual portal will fail.  The failure
   will be:
   
   com.ibm.wps.state.inputmediators.exceptions.VpAlreadyAssignedExc
   eption:
   EJPEI0118E: The ID [ObjectIDImpl '18_EHF14SD30G8QE02JRE6Q9F3531'
   [-1359524019530842578:2533519556720957568@0 /
   VIRTUAL_PORTAL_DEFINITION], Domain: [Domain: rel], DB
   representation:
   0000-2EBE40781B0022ED8098DB199DDE2823] of the virtual portal for
   the
   request cannot be assigned twice. The ID [ObjectIDImpl
   '18_EHF14SD30G8QE02JRE6Q9F3126'
   [-1359524019530842578:-4465074264212793216@0 /
   VIRTUAL_PORTAL_DEFINITION], Domain: [Domain: rel], DB
   representation:
   0000-2EBE40781B0022ED8098DB199DDE08C2] has already been assigned
   with
   the request com.ibm.ws.cache.servlet.CacheProxyRequest@6f2e5c8.
   
   
    
   
   

LOCAL FIX
 *  Clear cache in between two xmlaccess calls by restarting Portal.
   
   Alternatively, clear only:
   com.ibm.wps.datastore.services.Identification.OidAndUniqueName.c
   ache
   
   
    
   
   

PROBLEM SUMMARY
 *  When we execute the XMLAccess tool to generate the xml file
   of ExportRelease deck of base portal, virtual portal 's one
   after the another we noticed that the export of the second
   virtual portal fails with an error.
   
   
    
   
   

PROBLEM CONCLUSION
 *  Ensure cache to be correctly cleared after one XMLAccess
   command, so that the cache is clean for the next command.
