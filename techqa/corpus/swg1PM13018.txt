APAR STATUS
 * CLOSED AS PROGRAM ERROR.
    
   
   

ERROR DESCRIPTION
 *  DC unable to retrieve queue manager name for WAS/MQ stitching
   if queue manager name is not specified in queue connection
   factory definition or the queue manager name is null or blank
   when constructing com.ibm.mq.MQQueueManager object.
   Customer defined queue connection factory and queues at WAS
   cell/cluster level in order for their J2EE applications to use
   default queue manager on each physical aix box to send
   and receive messages (Binding Mode), for example,
   
   <factories
   xmi:type="resources.jms.mqseries:MQQueueConnectionFactory"
   xmi:id="MQQueueConnectionFactory_1251822128326" name="MQQCF"
   jndiName="MQQCF" description="This example
   QueueConnectionFactory uses
   the BINDINGS transport type to connect to the default Queue
   Manager on
   the local system. The properties specify the behaviour of the
   Pool of
   JMS Connections created from this Connection Factory. These
   properties
   specify the behaviour of the Pool of JMS Sessions created from
   this
   Connection Factory. This will produce XAQueueConnections."
   authMechanismPreference="BASIC_PASSWORD"
   authDataAlias="NU07-nc3sz1olbas11-dm/mqqcf" XAEnabled="false"
   port="0"
   transportType="BINDINGS" msgRetention="true">
   <connectionPool xmi:id="ConnectionPool_1251822128327"
   connectionTimeout="30" maxConnections="100" minConnections="1"
   reapTime="180" unusedTimeout="1800" agedTimeout="0"
   purgePolicy="FailingConnectionOnly"/>
   <mapping xmi:id="MappingModule_1251822128327"
   mappingConfigAlias="DefaultPrincipalMapping"
   authDataAlias="NU07-nc3sz1olbas11-dm/mqqcf"/>
   <sessionPool xmi:id="ConnectionPool_1251822128328"
   connectionTimeout="1800" maxConnections="10" minConnections="0"
   reapTime="10" unusedTimeout="1800" agedTimeout="1800"
   purgePolicy="EntirePool"/>
       </factories>
   For this type of configuration, current DC's callback methods
   could not obtain queue manager name so WAS transactions
   (sending or receiving messages) could not stitch with queue
   nodes generated by TT MQExits component.
   
   
    
   
   

LOCAL FIX

PROBLEM SUMMARY
 *  ****************************************************************
   * USERS AFFECTED:
   *
   * Applies to DC and TTAPI integration for end-to-end message
   * tracking for WebSphere when
   *  (1) using WebSphere MQ V6 libraries
   *  (2) Queue manager name is not specified in queue connection
   *      factory definition or the queue manager name is null
   *      or blank.
   *
   ****************************************************************
   *
   * PROBLEM DESCRIPTION:
   *
   * ITCAM for WebSphere DC is unable to retrieve queue manaager
   * name for WAS/MQ stitching if queue manager name is not
   * specified in queue connection factory definition or the
   * queue manager name is null or blank.
   *
   ****************************************************************
   
   
    
   
   

PROBLEM CONCLUSION
 *  This fix has introduced the following:
   
   (1) MQ callbacks has been enhanced with MQINQ call to retrieve
   default queue manager name.
   
   (2) A new property is introduced so clients can specify default
   queue manager name to avoid MQINQ call:
   
      am.dc.mqi.default.qmgr.name
   
     This property can be set in
   <DC_HOME>/runtime/<server>/custom/toolkit.properties file.
   
   (3) In order to lower overhead, another property (default value
   set to true) is introduced so MQ callback code can cache default
   queue manager name (from MQINQ call):
   
       am.dc.mqi.cache.default.qmgr.name
   
   
   The fix for this APAR is contained in the following maintenance
   packages:
   
     | interim fix | 6.1.0.5-TIV-ITCAMfWAS_MP-IF0011
   
     | fix pack | 7.1.0-TIV-ITCAMAD-FP0001
