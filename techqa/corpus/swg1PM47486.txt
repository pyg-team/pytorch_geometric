APAR STATUS
 * CLOSED AS PROGRAM ERROR.
    
   
   

ERROR DESCRIPTION
 *  Due to an exception thrown from the transaction extension helper
   when a global transaction is to be suspended, it is possible
   that the global transaction stays suspended and is never
   resumed. This causes connections to remain in connection pools
   that are associated with this suspended transaction. This
   ultimately causes connection pool exhaustion and hanging threads
   when connections are attempted to be retrieved from the, now
   exhausted, connection pool.
   
   A connection pool trace will indicate connections are in the
   following state...
   
   Connection marked to be destroyed.  Waiting for transaction end
   and connection close -
   com.ibm.ws.Transaction.JTA.TransactionImpl@2c2c2c2c#tid=70429
   MCWrapper id 20882088  Managed connection
   WSRdbManagedConnectionImpl@36333633
   State:STATE_TRAN_WRAPPER_INUSE Thread Id: 00000343 Thread Name:
   Default: 30 Handle count 1 Start time inuse Tue Aug 23 15:56:17
   BST 2011 Time inuse 788 (seconds)
   
   
    
   
   

LOCAL FIX

PROBLEM SUMMARY
 *  ****************************************************************
   * USERS AFFECTED:  All users of IBM WebSphere Application      *
   *                  Server                                      *
   ****************************************************************
   * PROBLEM DESCRIPTION: Global transaction is not resumed       *
   *                      after scheduler service attempts to     *
   *                      start a new local transaction fails     *
   *                      with an exception.                      *
   ****************************************************************
   * RECOMMENDATION:                                              *
   ****************************************************************
   When an application invokes a scheduler service with in a global
   transaction context, it will get suspended as the scheduler
   attempts to start a new local transaction for its own execution.
   At times when the scheduler's attempt to start a new local
   transaction fails, the global transaction is never resumed. This
   behavior leads to numerous issues such as connection leakage,
   application hung etc..
   
   
    
   
   

PROBLEM CONCLUSION
 *  The code has been fixed to resume the global transaction, if
   any, as a cleanup process.
   
   The fix for this APAR is currently targeted for inclusion in
   fix pack 6.1.0.43, 7.0.0.23 and 8.0.0.3.  Please refer to the
   Recommended
   Updates
