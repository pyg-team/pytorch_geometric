APAR STATUS
 * CLOSED AS PROGRAM ERROR.
    
   
   

ERROR DESCRIPTION
 *  A web service client in WebSphere Application Server tries to
   make an SSL connection to a 3rd party (non-WebSphere
   Application Server) web service provider.  The first connection
   (first web service call) always works.  The second and
   subsequent connections (web service calls) always fail.
   
   ----------------------------------------------------------------
   --------
   Failing case:  A cached session is used to attempt the SSL
   connection.
   Note that the 'getSSLContext_JSSE2' method call returns a cache
   hit in the failing case.  The remote server side (3rd party web
   service provider) does NOT support cached connections.
   
   
    
   
   

LOCAL FIX
 *  No workaround
   
   
    
   
   

PROBLEM SUMMARY
 *  ****************************************************************
   * USERS AFFECTED:  All users of IBM WebSphere Application      *
   *                  Server V6.1 and V7.0                        *
   ****************************************************************
   * PROBLEM DESCRIPTION: The 3rd party web service provider      *
   *                      does not support cached SSL             *
   *                      connections. Cached SSL connections     *
   *                      are the default behavior for            *
   *                      WebSphere Application Server            *
   ****************************************************************
   * RECOMMENDATION:                                              *
   ****************************************************************
   The default behavior for WebSphere Application Server is to
   use cached SSL connections. Since the third party web service
   does not support SSL connections, the initial outbound
   connection would be successful, and all subsequent connections
   will be rejected by the third party web service because each
   subsequent connection uses cached session data.
   
   
    
   
   

PROBLEM CONCLUSION
 *  The SSL connection code was enhanced to support a new JVM
   system property: "com.ibm.ws.ssl.skipSSLContextCache". This
   property, when set to true, will cause each outbound SSL
   connection to not use any cached connection data, treating each
   connection as a new SSL connection.
   
   The fix for this APAR is currently targeted for inclusion in
   fix packs 6.1.0.35 and 7.0.0.13.  Please refer to the
   Recommended Updates page for delivery information:
   http://www.ibm.com/support/docview.wss?rs=180&uid=swg27004980 [http://www.ibm.com/support/docview.wss?rs=180&uid=swg27004980]
