APAR STATUS
 * CLOSED AS PROGRAM ERROR.
    
   
   

ERROR DESCRIPTION
 *  WAS / JazzSM seems to hang periodicly during creation of new
   dahboards.  Only workaround is to restart WAS.
   
   Even after adjusting JVM heap size to 1.5GB, there was still a
   need to recycle WAS once a week because it would hange and
   nothing would work in the TSAAM console any more. It wasn't
   even possible to stop the components normally.
   
   Analysis of javacore found that ActiveMQ threads were parked
   and blocking other threads. ActiveMQ is being used by
   DASH/JazzSM.
   
   
    
   
   

LOCAL FIX
 *  Restart WAS frequently
   
   
    
   
   

PROBLEM SUMMARY
 *  ****************************************************************
   * USERS AFFECTED: Tivoli System Automation Application Manager
   * (SA AM) 4.1.0.0 users
   ****************************************************************
   * PROBLEM DESCRIPTION:
   * WAS / JazzSM seems to hang periodically during creation of
   * new dahboards. This can only be solved by restarting WAS.
   * Adjusting (increasing) the JVM heap size to 1.5GB does not
   * solve the issue.
   * Issue is caused by ActiveMQ threads that were parked and
   * blocking other threads. ActiveMQ is being used by DASH/JazzSM.
   * JazzSM support identified those problems are addressed within
   * JazzSM 1.1.2, but SA AM 4.1.0.0 cannot use this JazzSM level
   * without a toleration fix.
   ****************************************************************
   * RECOMMENDATION:
   ****************************************************************
   
   
    
   
   

PROBLEM CONCLUSION
 *  The internal handling of the SA AM has been adjusted to work
   with JazzSM 1.1.2 . Installation of JazzSM 1.1.2 and this
   toleration fix addresses the observed hang problems.
   .
   The official fix will be included within fix pack 1 of Tivoli
   System Automation Application Manager 4.1.0 .
