APAR STATUS
 * CLOSED AS PROGRAM ERROR.
    
   
   

ERROR DESCRIPTION
 *  The users has to re-authenticate very often, and before the
   timeout values reached.
   
   following settings in webseald.conf
   [session]
   max-entries = 4096
   timeout = 3600
   inactive-timeout = 0
   
   When inactive-timeout = 0, the control of cache entries is then
   governed by the timeout and max-entries stanza entries. So
   WebSeal will store up to 4096 credentials in the cache. And
   when a new credential arrive an old will removed.
   
   Customer had retested it in his test environment with only few
   users working on it and collected statistics. After first login
   pdadmin sec_master> server task ceeuat-webseald-pmdl02 stats get
   pdweb.sescache
   hit      : 26
   miss     : 2
   add      : 1
   del      : 0
   inactive : 0
   lifetime : 0
   LRU expired : 0
   
   Then he refreshs after 10 minutes and was asked to
   re-authenticate by WebSeal. Here the statistics:
   pdadmin sec_master> server task ceeuat-webseald-pmdl02 stats get
   pdweb.sescache
   hit      : 27
   miss     : 4
   add      : 2
   del      : 0
   inactive : 0
   lifetime : 0
   LRU expired : 0
   
   
   In the pdweb.snoop trace I can see that client contacts WebSeal
   with the same cookie as 10 minutes ago, but WebSeal returns with
   a login request.
   
   
   recreations steps:
   1 - login to WebSeal and access a page on backend
   2 - wait for 10 minutes or little more
   3 - refresh the page within your browser
   4 - WebSeal will prompt for re-authentication
   
   
    
   
   

LOCAL FIX
 *  Set inactive-timeout other than zero. For example same as
   timeout value.
   
   
    
   
   

PROBLEM SUMMARY
 *  WebSEAL does not honor the  session  'timeout'
   value approproriately when inactive-timeout=0, WebSEAL inadverte
   ntly expires a login session before reaching the timeout value.
