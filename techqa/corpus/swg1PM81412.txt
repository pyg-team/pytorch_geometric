APAR STATUS
 * CLOSED AS PROGRAM ERROR.
    
   
   

ERROR DESCRIPTION
 *  A JAX-WS web service provider running within WebSphere
   Application Server may intermittently send SOAP response
   messages to the web service client that are missing a namespace
   within an element.  For example:
   
   Expected element:
   <item xsi:type="ns2:myItem"
   xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">
   
   Incorrect element without the namespace:
   <item xsi:type="locationReferenceItem"
   xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">
   
   
    
   
   

LOCAL FIX

PROBLEM SUMMARY
 *  ****************************************************************
   * USERS AFFECTED:  All users of IBM WebSphere Application      *
   *                  Server using Axis2 as WebServices server    *
   ****************************************************************
   * PROBLEM DESCRIPTION: WebServices axis2 response is missing   *
   *                      namespace prefix intermittently at      *
   *                      "xsi:type" element                      *
   ****************************************************************
   * RECOMMENDATION:                                              *
   ****************************************************************
   Normally at client side we get an axis2 good response like:
   <item xsi:type="ns2:locationReferenceItem"
   xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">
   However, after the server has been recycling for couple of
   days, the wrong response may return:
   <item xsi:type="locationReferenceItem"
   xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">
   Payload is not appended with the namespace prefix. Sometimes,
   a ClassCastException may occur as well.
   The issue will disappear after the server restarts.
   
   
    
   
   

PROBLEM CONCLUSION
 *  In IBM Axis2, there are two methods that will invoke
   JAXBContext generation. The first method uses the correct
   webs ervice namespace to generate JAXBContext, the other one
   does not. JAXBContext is cached by SoftReference, which may be
   collected by the JVM if there is not enough system memory.
   When the softreference with the JAXBContext is collected by the
   jvm, a new JAXBContext needs to be constructed. If the new
   JAXBContex is generated in the first method, the issue would
   not appear. However, if the new JAXBContex is created in the
   second method, the problematic JAXBContext would be created.
   During marshalling, the response would lose the namespace
   prefix. The second method has been corrected.
   
   The fix for this APAR is currently targeted for inclusion in
   fix packs 7.0.0.31, 8.0.0.7 and 8.5.5.1.
   Please refer to the Recommended Updates page for delivery
   information:
   http://www.ibm.com/support/docview.wss?rs=180&uid=swg27004980 [http://www.ibm.com/support/docview.wss?rs=180&uid=swg27004980]
