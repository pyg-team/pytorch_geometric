APAR STATUS
 * CLOSED AS PROGRAM ERROR.
    
   
   

ERROR DESCRIPTION
 *  Problem Description: Customer is using TAM 6.0 policy server
   with TAM api's 5.1. They have specified 4 entries for
   appsvr-authzsvrs in the PDPerm.properties file. If the first
   authorization is down, the 2nd one specified is used and api
   application works fine. However, if the first two authorization
   servers are down, the api fails. The configuration in the config
   file looks okay and customer has performed the following tests:
   
   
   TEST AUTH server individually with no issue:
   1) apsvr-authzsvrs=AUTH1:7136:1,  API call success
   2) apsvr-authzsvrs=AUTH2:7136:1,  API call success
   3) apsvr-authzsvrs=AUTH3:7136:1,  API call success
   4) apsvr-authzsvrs=AUTH4:7136:1,  API call success
   
   TEST multiple AUTH server:
   1)
   apsvr-authzsvrs=AUTH1:7136:1;AUTH2:7136:1;AUTH3:7136:1;AUTH4:713
   6:1;
   All AUTH servers are up, API call success
   
   2)
   apsvr-authzsvrs=AUTH1:7136:1;AUTH2:7136:1;AUTH3:7136:1;AUTH4:713
   6:1;
   AUTH1 down, others are up, API call success
   
   3)
   apsvr-authzsvrs=AUTH1:7136:1;AUTH2:7136:1;AUTH3:7136:1;AUTH4:713
   6:1;
   AUTH1 and AUTH2 down, others are up, API call failed with
   'Cannot
   contact
   server' message
   
   
   I have collected the PDPerm.properties file and customer is
   performing the scenario and collecting JLog tracing.
   
   
    
   
   

LOCAL FIX

PROBLEM SUMMARY
 *  When specifying more than 2 entries for appsvr-
   authzsvrs in the PDPerm.properties file and first two are down o
   r otherwise inaccessible, the failover does not use the 3rd serv
   er or beyond.

   ained in the following maintenance delivery vehicles:

   | fix pack | 5.1.0-TIV-TAM-FP0034
