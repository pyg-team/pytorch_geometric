APAR STATUS
 * CLOSED AS PROGRAM ERROR.
    
   
   

ERROR DESCRIPTION
 *  It has been reported by customers and since reproduced by IBM
   that under certain circumstances, the same event can get updated
   by a policy more than once in a single second. The statechange
   does not reflect multiple changes and the filter then fails to
   catch the event during the next event reader poll. The
   underlying cause has been found to be due to a portion of the
   logic code used to check for new events in the filter buffer.
   
   
    
   
   

LOCAL FIX
 *  There are a couple of options for a workaround. The initial
   workaround is currently being used by the customer. A trigger
   has been put in place on the object server to check for any
   events that match the filter query and are older than 30
   seconds. If a match is found, the event is "touched" and the new
   timestamp causes the query filter in Impact to gather the event.
   Another workaround would be to ammend the policy to touch the
   Tally value (Tally=@Tally+1).
   
   
    
   
   

PROBLEM SUMMARY
 *  Impact does not process events if the event is updated within
   the same second that it was last updated
   
   
    
   
   

PROBLEM CONCLUSION
 *  Impact compares the StateChange, Tally, ServerName and
   ServerSerial to mark events as unique.  If all of these fields
   are the same, I made impact check the rest of the fields to
   verify that the event is not unique
   
   The fix for this APAR is contained in the following maintenance
   package:
   |Interim Fix| 5.1.1-NCI-IF0003
